Sensor Data Processing

Orchestrated with Apache Airflow, PostgreSQL, and MLflow for robust real-world data engineering.

"

The goal of this project was to design an ETL pipeline using Apache Airflow that simulates a real-world data processing workflow. Instead of training machine learning models inside Airflow (which is not recommended), the pipeline focuses on data ingestion, transformation, and analytics processing.

Core Functional Tasks:

  • Simulate raw sensor data and store it in a PostgreSQL database.
  • Extract the dataset from PostgreSQL.
  • Perform transformations (Aggregations, Mean calculations, Trend slope analysis).
  • Store the transformed dataset in another PostgreSQL table.
  • Log dataset lineage, metrics, and metadata using MLflow.
  • Provide observability through logs and notifications.

System Architecture

The system consists of Apache Airflow for orchestration, PostgreSQL for storage, MLflow for tracking, and Python for data processing. The workflow ensures a smooth data flow from generation to notification.

Implementation Strategy: The "Why's"

A breakdown of the critical design decisions made during the construction of this pipeline.

1. Append vs. Replace

Decision: Use Append loading mode for all target tables.

The Why: Replacing data loses historical context. Appending allows us to build a rich time-series dataset, essential for detecting long-term sensor drift and seasonal trends.

2. Metadata Injection

Decision: Inclusion of run_date and batch_id in every record.

The Why: This enables atomic traceability. We can identify exactly which Airflow execution produced a specific row, making data reconciliation and partitioned reprocessing trivial.

3. Circuit-Breaker Validation

Decision: Strict Pydantic models before transformation.

The Why: "Garbage in, garbage out." By validating sensor ranges at the ingestion stage, we prevent faulty readings from bloating MLflow metrics or skewing aggregation slopes.

4. Unsupervised AI

Decision: Isolation Forest for anomaly detection.

The Why: Real-world sensor data rarely has labels. Isolation Forest works out-of-the-box on unlabeled distributions, flagging outliers based on statistical sparsity alone.

Pipeline Tasks

🏗️

Task 1: Database Setup

setup_database_schema()

Creates the required PostgreSQL tables if they do not exist, ensuring schema availability and preventing failures.

📡

Task 2: Raw Data Generation

create_and_dump_raw_data()

Simulates sensor readings for the previous 24 hours (temperature, humidity, pressure) and inserts them into raw_sensor_data.

⚡

Task 3: ETL Processing

extract_transform_load_and_log()

1. Extract & Validate

Data extracted from PostgreSQL and validated using Pydantic (Temp: -50 to 100, Humidity: 0 to 100, Pressure: 800 to 1200).

2. Anomaly Detection

Uses Isolation Forest (contamination=0.03) to detect abnormal readings, adding the is_anomaly label.

3. Feature Engineering

Creation of hour and heat_index (temperature + 0.5 * humidity).

4. Aggregation & Trend

Groups analytics and calculates temperature trend slope using numpy.polyfit() for linear regression.
